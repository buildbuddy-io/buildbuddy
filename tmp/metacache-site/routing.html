Routing — finding the right range & replica

The system is multi-group raft: every key range is its own dragonboat cluster. A request keyed by some byte-string must (a) discover which range owns the key, (b) discover where that range's replicas live, (c) try them in preference order, and (d) handle stale routing information when ranges have split or moved.

Sender — the request router

The Sender is the single client-side entry point for "do this batch on key K." It hides multi-group raft from callers and presents a uniform interface.

}

type Sender struct {
    rangeCache *rangecache.RangeCache  // local cache: key range -> RangeDescriptor
    apiClient  *client.APIClient       // pooled gRPC connections to peers
}

Finding the range for a key

  1. Cache lookup. Sender.LookupRangeDescriptor(ctx, key, skipCache) first tries s.rangeCache.Get(key). The rangecache uses an interval tree to find the unique range whose [start, end) contains the key.
  2. On miss (or skipCache), call fetchRangeDescriptorsFromMetaRange(ctx, [][]byte{key}). This executes a ScanRequest on the meta range using RangeMetaKey(key) = \x02 || key with SEEKGT_SCAN_TYPE, limit 1 — "find the descriptor whose meta-key is smallest >= my key's meta-key." Verifies the returned descriptor actually contains key (rangemap.Range.Contains).
  3. Cache the result via rangeCache.UpdateRange. Update only if newGeneration > existingGeneration.

The meta range itself is discovered via gossip. Sender.GetMetaRangeDescriptor() just reads the rangecache, which is seeded by the MetaRangeTag serf user-event published by every node hosting the meta range.

Trying replicas in preference order

tryReplicas iterates rd.Replicas (then rd.Staging) and runs the supplied closure against each. The interesting bit is the error classification:

sender.go:307-355 — tryReplica error handling
if status.IsOutOfRangeError(err) {
    switch m := status.Message(err); {
    case strings.HasPrefix(m, constants.RangeNotCurrentMsg):
        // Range descriptor itself is stale; no replica can help. Bail.
        return false, status.OutOfRangeErrorf(...)
    case strings.HasPrefix(m, constants.RangeNotLeasedMsg),
         strings.HasPrefix(m, constants.RangeLeaseInvalidMsg),
         strings.HasPrefix(m, constants.RangeNotFoundMsg):
        // This replica is wrong; try the next one.
        return true, status.WrapErrorf(...)
    }
}
if status.IsUnavailableError(err) {
    return true, ...   // network problem - try another replica
}

On success, the index of the winning replica is returned so the caller can call rangeCache.SetPreferredReplica — promoting that replica to index 0 in the cached rd.Replicas so future calls hit the leaseholder first.

The main retry loop

sender.go:443 — Sender.run
retrier := retry.DefaultWithContext(ctx)
skipRangeCache := false
for retrier.Next() {
    rd, err := s.LookupRangeDescriptor(ctx, key, skipRangeCache)
    if err != nil { ...; continue }
    i, err := s.tryReplicas(ctx, rd, fn, opts.ConsistencyMode)
    if err == nil {
        if i != 0 {
            s.rangeCache.SetPreferredReplica(ctx, rd.Replicas[i], rd)
        }
        return nil
    }
    skipRangeCache = true   // next iteration goes back to the meta range
    if !status.IsOutOfRangeError(err) && !isConflictKeyError(err) {
        return err
    }
    lastError = err
}

Once we see an OutOfRange or ConflictKey error, skipRangeCache = true forces a fresh meta-range lookup on the next iteration — that's how stale routing self-heals.

Multi-key fan-out

RunMultiKey (sender.go:552) takes N keys and runs one closure per destination range, in parallel:

  1. partitionKeysByRange: cache lookup for each; batch-fetch misses with a single meta-range scan; group by range_id. Detects mid-flight changes (two RDs for the same range_id after a split) and retries the partitioning.
  2. Fan out via errgroup with SetLimit(100); each goroutine runs tryReplicas.
  3. Per-range outcome: success → collect response; OutOfRange → re-queue those keys, retry with fresh cache; other error → abort.

Registry — NHID → address

The registry tracks two distinct mappings:

  1. (rangeID, replicaID) → nhid — for dragonboat's transport layer.
  2. nhid → {raftAddr, grpcAddr} — for both dragonboat and the application's gRPC client.

StaticRegistry

In-memory map. Satisfies dragonboat's raftio.INodeRegistry interface: Add, Remove, RemoveShard, Resolve(rangeID, replicaID) → (addr, connectionKey, error). Plus app-facing ResolveGRPC(nhid), AddNode(nhid, raftAddr, grpcAddr), ListNodes().

The streamConnections flag controls multiplexing — when >1, getConnectionKey returns "addr-partitionID" so dragonboat opens that many physical TCP connections per peer for snake-shaped raft traffic.

DynamicNodeRegistry — gossip-backed

Production wrapper around StaticRegistry. Two gossip channels:

Push (broadcast)

When local AddNode happens, gossip a RegistryPushRequest{nhid, raft, grpc} user-event. All listeners' handleEvent ingests into their local StaticRegistry.

Pull (query/response)

On ResolveGRPC/Lookup miss, issue a gossip RegistryQueryEvent. Peers' handleQuery look it up; first response wins; caller adds to local StaticRegistry.

Why not gossip the (rangeID, replicaID) mapping too?
DynamicNodeRegistry.Add deliberately does not gossip per-replica mappings because dragonboat calls this very frequently (every shard membership change for every shard). Instead, the mapping is reconstructed on demand via gossip query and then cached. This keeps gossip traffic O(nodes) rather than O(replicas).

Optional Kubernetes pod watcher

If a PodWatcherManager is set, raft addresses that look like pod FQDNs are watched via the K8s API. When a pod's IP changes, resolvedRaftAddrs is updated and Resolve() returns the new IP — which dragonboat uses as a new connection key, triggering a reconnect. This is the IP-tracking sister to the kube:/// gRPC resolver in client.go.

RangeCache — local descriptor cache

An interval tree (rangemap.RangeMap) keyed by [start, end). Each entry wraps a *RangeDescriptor in its own mutex so the replica list can be reordered (for preferred-replica promotion) without holding the map-level lock.

How it's populated

  1. Gossip seeding for the meta range. Store.setMetaRangeBuf is called from gossip events. After parsing the descriptor, calls sender.UpdateRange(new). This is how the chicken-and-egg meta-range bootstrap works.
  2. Cache misses fetched from the meta range. LookupRangeDescriptor and partitionKeysByRange call rangeCache.UpdateRange.
  3. Preferred-replica promotion. SetPreferredReplica reorders the cached replica list after a successful call.

How it's invalidated

No TTL. Updates only happen if newGeneration > existingGeneration. Splits/merges that change range bounds use AddAndRemoveOverlapping which atomically replaces older-generation entries.

Implicit invalidation via stale-range errors: when Sender.run sees OutOfRange, it sets skipRangeCache = true and re-fetches. The fresh descriptor's higher generation will replace the stale entry on the next UpdateRange.

RBuilder — request builder

A fluent builder for BatchCmdRequest. The BatchCmdRequest is the universal envelope for everything (reads, writes, admin, txn control).

// Type-switches on the proto type and wraps in RequestUnion.
func (b *BatchBuilder) Add(req interface{}) *BatchBuilder { ... }

Other features:

  • SetTransactionID(txid) — marks the batch as part of a 2PC; replica parks the batch until COMMIT or ROLLBACK.
  • SetFinalizeOperation(op) — turns the batch into the second 2PC phase.
  • SetHeader(h) / SetSession(s) — routing + idempotency.
  • SetLockMappedRange(lock) — for split prepare (locks the whole left range against new writes).
  • AddPostCommitHook(SnapshotClusterHook | StartShardHook) — side-effects after the entry applies.

The corresponding BatchResponse has typed getters per slot, plus an AnyError() that checks per-op statuses (because batch-level success and per-op success are distinct).

Storemap — driver's view of the cluster

Not used by routing. StoreMap is the cluster-wide view of every node's load, capacity, and aliveness, used by the driver (covered separately) to decide where to place new replicas and where to move leases. Mentioned here because it lives alongside the routing infrastructure.

Two mechanisms keep it fresh: (a) gossip member events update each node's StoreUsage tag; (b) every read does a live gossipManager.Members() query so aliveness is fresh. Status state machine: Available / Suspect (alive but recently recovered) / Unknown (down <5m) / Dead (down ≥5m).
