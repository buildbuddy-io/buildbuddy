MetaCache interfaces.Cache

metacache.Cache is a thin, mostly stateless adapter. It receives Bazel-cache-shaped requests (digests, instance names, compressor preferences) and turns each one into one metadata RPC to the raft cluster plus, when the blob isn't small enough to inline, one blob I/O to GCS. All the durability and consensus complexity is somewhere else.

File
enterprise/server/backends/metacache/metacache.go (867 lines). Tests at metacache_test.go.

Why split metadata from bytes?

Two reasons drive the split:

  1. Eviction by GCS lifecycle. Each GCS object is written with a customTime equal to "now." The bucket's lifecycle policy deletes objects with DaysSinceCustomTime > N. The metadata record stores the same last_custom_time_usec so a read can detect "this object is about to vanish" before hitting GCS.
  2. Cost optimization for small files. Objects below MaxInlineFileSizeBytes (default 1 024 bytes) skip GCS entirely and ride along in the metadata record's StorageMetadata.InlineMetadata. This is most CAS entries in practice.

Construction & configuration

Registered via cache.meta_cache.* flags, dispatched to cache_config.MetaCacheConfig. If MetadataBackend is empty, Register short-circuits and does nothing.

In New (metacache.go:160-203) the GCS-backed filestore.Store is constructed with GCS-level compression disabled (metacache does its own zstd, so doubling up would just burn CPU). It then calls SetBucketCustomTimeTTL(ctx, opts.GCSTTLDays) and refuses to start if that fails — the TTL is load-bearing.

GCS TTL is load-bearing
Metacache pre-checks ObjectIsPastTTL with a 1-hour safety buffer (gcsutil.go) on every read. The buffer covers the gap between GCS lifecycle marking and actual deletion, which run as decoupled jobs.

Write path, end-to-end

Let's trace a single Set(ctx, r, data) all the way through:

  1. Set at metacache.go:640-653 calls writerWithImmediateCommit(ctx, r, len(data)), writes, commits, closes. Defers Close before Commit on purpose — the GCS wrapper's Close() only cancels a context, not the upload.
  2. writerWithImmediateCommit (:438) installs a writeMetadataFn that calls setMetadatas(ctx, md) the moment the bytes have committed.
  3. writer (:452) is where the compression decision happens: 
    shouldCompress := r.GetCompressor() == repb.Compressor_IDENTITY &&
                  sizeHint >= c.opts.MinBytesAutoZstdCompression
if shouldCompress {
    r = &rspb.ResourceName{ ..., Compressor: repb.Compressor_ZSTD }
}
fileRecord, _ := c.makeFileRecord(ctx, r)
    Then newWrappedWriter picks the storage backend.
  4. writerForRecord (:377-387) — the inline-vs-blob switch on size hint: 
    if sizeHint < c.opts.MaxInlineFileSizeBytes {
    return c.opts.FileStorer.InlineWriter(ctx, sizeHint), nil
}
return c.opts.FileStorer.BlobWriter(ctx, fileRecord)
    Inline returns a bytes.Buffer-backed MetadataWriteCloser; blob returns a GCS CommittedMetadataWriteCloser.
  5. newWrappedWriter (:389-431) wraps with ioutil.NewCustomCommitWriteCloser and installs the commit function that assembles the FileMetadata after bytes are committed: 
    cwc.SetCommitFn(func(bytesWritten int64) error {
    now := c.opts.Clock.Now().UnixMicro()
    md := &sgpb.FileMetadata{
        FileRecord:         fileRecord,
        StorageMetadata:    wcm.Metadata(),     // inline bytes OR gcs blobName+customTime
        EncryptionMetadata: encryptionMetadata,
        StoredSizeBytes:    bytesWritten,
        LastAccessUsec:     now,
        LastModifyUsec:     now,
    }
    if bytesWritten == 0 { return status.UnavailableError("zero-length writes are not allowed") }
    return fn(md)  // the caller-supplied writeMetadataFn
})
    If encryption is enabled, the writer chain gets wrapped in the crypter's encryptor here too.
  6. zstd wrap (if shouldCompress) via compression.NewZstdCompressingWriteCommiter, drawing 4 MiB buffers from c.bufferPool. Each 4 MiB chunk is compressed independently.
  7. Caller writes bytes → flows through compression → encryption → inline buffer or GCS streaming writer.
  8. Caller calls Commit(): compression flushes (records compression-ratio metric), encryption flushes, GCS conditional writer commits. For inline this is a no-op.
  9. After inner commit succeeds, the wrapper's CommitFn runs: builds the FileMetadata, calls setMetadatas.
  10. setMetadatas issues one MetadataService.Set gRPC. From here onward we're in the MetadataService page.
Two-phase, but not atomic
The order is strictly (1) bytes, then (2) metadata. There is no transaction:
  • Bytes succeed + metadata fails → blob orphaned, eventually reaped by TTL.
  • Metadata exists but blob missing → next read detects via handleMetadataMismatch and conditionally deletes the metadata.

Read path, end-to-end

Reader(ctx, r, offset, limit) (:796) does the obvious: one Get RPC, then open the backend reader from the returned StorageMetadata. The internal reader (:705-793) is where the matrix-of-cases lives.

Decisions made in reader()

  1. Zero-length guard (stored_size_bytes == 0 → NotFound).
  2. GCS TTL pre-check (ObjectIsPastTTL with 1h buffer → NotFound).
  3. Reject compressed passthrough with offset/limit.
  4. If encrypted but encryption now disabled → NotFound.
  5. Decide rawStorage = (cached==IDENTITY && !decrypt); if true, push offset/limit to the backend.
  6. Open backend reader (inline or GCS). NotFound → handleMetadataMismatch.
  7. Wrap with decryptor, then decompressor, then offset skip, then limit.
  8. If cached IDENTITY but client wants ZSTD, compress on read.

GCS interactions

Object keys are constructed by filestore.blobKey (filestore.go:684-707): 

// {appName}/{partitionPrefix+partitionID}/{groupID}/{ac|cas}/{hashPrefix:4}/{hash}
// e.g.  buildbuddy-app-0/PART_default/GR123/cas/abcd/abcd12345…
if r.GetEncryption().GetKeyId() != "" {
    hash += "_" + r.GetEncryption().GetKeyId()
}

Then BlobWriter (filestore.go:866-900) appends a random salt to the blobName: blobName + "-" + random.RandomString(5). So two writes of the same digest produce two distinct GCS objects. The metadata service's row authoritatively says which one is current; stale ones get TTL'd. customTime = now is set on every write; the GCS lifecycle rule uses DaysSinceCustomTime.

Note: metacache.Delete only deletes the metadata. GCS object cleanup is delegated to the bucket lifecycle policy.

Notable subtleties

"Empty FileMetadata = NotFound" sentinel

MetadataService.Get doesn't have an explicit NotFound — it returns an empty FileMetadata proto in the slot. Proto round-trip turns "nil" into a non-nil zero struct, so the metacache checks md.GetFileRecord() == nil (see :819-826). Fragile: if the metadata service ever sets FileRecord to a zero-but-non-nil proto, this would misdetect.

handleMetadataMismatch — the dangling-pointer cleanup

When the backend reader returns NotFound on a GCS blob, handleMetadataMismatch (:339-365) calls deleteFileAndMetadata with match_atime = md.LastAccessUsec as an optimistic-concurrency guard. If another writer wrote a fresher record concurrently, the delete fails with FailedPrecondition and the new record is left alone. For inline records, mismatch is impossible-by-construction; the code logs "Impossible metadata mismatch: something is broken."

SetMulti's mutex-shared mds slice

Each parallel writer's commit fn appends to a shared mds []*FileMetadata under a mutex. After eg.Wait(), one setMetadatas(ctx, mds...) call ships all the metadata in a single RPC. Trades a small amount of latency (wait for slowest write) for a huge reduction in RPC count. See :655-691.

GCS write race tolerance

Despite the random salt, gcsMetadataWriter.Commit swallows AlreadyExists and ResourceExhausted ("too many writes to this object"). Both mean "some valid write of equivalent bytes happened" — treating as success is safe (filestore.go:830-847).

Inline reads serve the same bytes over and over

Each InlineReader call gets fresh bytes from the metadata service. For a hot inline blob, the metadata service eats the load. There is no local-side caching.

GetMulti's silent skip on NotFound

Per-item NotFound/os.IsNotExist → silent continue (the digest just won't appear in the result map). Other errors abort the entire GetMulti.

Compressor field copying is fragile

When auto-zstd engages, the writer manually copies ResourceName fields to set Compressor: ZSTD. If a new field is added to ResourceName, it has to be added here too. See :466-472.

Concurrency notes

  • Goroutines: only SetMulti spawns goroutines (errgroup with SetLimit(MaxWriteGoroutines)). GetMulti is sequential — a potential optimization opportunity.
  • Locks: the only mutex is local to SetMulti, protecting the shared mds slice.
  • Connection pooling: the metadata gRPC client is dialed with pool size 2 (grpc_client.DialInternalWithPoolSize(env, opts.MetadataBackend, 2)).
  • Buffer pool: a single bytebufferpool.VariableSizePool(4MB) is shared across all concurrent reads/writes; safe for concurrent use.
  • Tracing & metrics: every public method uses a defer recordMetrics(...) with start time; both CacheMethodHandledTotal and CacheMethodHandlingUsec are emitted, labeled by status.
