Store & Replica

The dragonboat integration. Store is the per-node orchestrator that owns the NodeHost, the Pebble DB, and every supporting component. Replica is the dragonboat IOnDiskStateMachine implementation that turns committed raft entries into Pebble writes.

Files
enterprise/server/raft/store/store.go (4 226 lines, the biggest file in the system).
enterprise/server/raft/replica/replica.go (2 058 lines).
Plus small support files: listener.go, events.go, constants.go, keys.go, header.go, config.go, logger.go.

What dragonboat gives us

Dragonboat (github.com/lni/dragonboat, v4) is a Go library for Multi-Group Raft: many independent raft state machines hosted in one process, sharing a transport and log database. BuildBuddy uses dragonboat's:

  • NodeHost — one per process. Runs the transport, hosts all raft state machines, sets up the log DB.
  • statemachine.IOnDiskStateMachine interface — the user-provided state machine. BuildBuddy's Replica implements it directly.
  • raftio.INodeRegistry — pluggable address resolver. BuildBuddy supplies a gossip-backed one (the Registry).
  • RaftEventListener + SystemEventListener — observe leader changes, node ready, snapshot events. BuildBuddy's RaftListener adapts them to in-process channels.
  • Snapshotting machinery — dragonboat decides when to snapshot; the state machine provides the bytes.
  • Membership operations — SyncRequestAddReplica, SyncRequestAddNonVoting, SyncRequestDeleteReplica, RequestLeaderTransfer, StartOnDiskReplica.
  • Read/write API — SyncPropose, SyncRead (ReadIndex), StaleRead.

What it does not give us (BuildBuddy builds on top): range descriptors, range leases, multi-range transactions, placement decisions, idempotency sessions, address discovery.

One Store per node. Its job is to host every replica that lives on the node and orchestrate everything else.

Field tour
  • nodeHost: the dragonboat handle. Created in New() via dragonboat.NewNodeHost(nhc).
  • gossipManager: serf-based discovery. Used for meta-range descriptor propagation, per-node usage tags, and bringup coordination.
  • sender: client-side request router. See Routing.
  • registry: gossip-backed INodeRegistry that translates (rangeID, replicaID) → NHID → (raftAddr, grpcAddr).
  • apiClient: pooled gRPC connections to peers; calls raft.service.Api.
  • liveness: epoch-based node-liveness; renewed in meta-range. Anchors range leases. See Leases.
  • Sessions: four distinct client.Sessions separate traffic classes so eviction RPCs don't starve user writes.
  • db: one Pebble DB at <rootDir>/raft_store/ shared by every replica on this node.
  • openRanges + rangeMap: two views of "what ranges are on this node." Map by ID; interval tree by [start, end).
  • leaseKeeper: per-range lease management.
  • replicas sync.Map: rangeID → *replica.Replica.
  • events: fan-in channel for range-level events; handleEvents dispatches.
  • driverQueue: the placement driver's priority queue.
  • txnCoordinator: 2PC for multi-range ops.

Lifecycle

  1. New(env, cfg, opts...) at store.go:281-488. Builds the listener, the registry holder, then dragonboat.NewNodeHost(nhc). Opens Pebble. Constructs leasekeeper, usagetracker, txn coordinator, driverQueue. Starts the gRPC server. Critically: re-opens every previously-known replica by iterating nodeHost.GetNodeHostInfo().LogInfo and calling StartOnDiskReplica for each, up to numReplicaStarter=50 in parallel.
  2. Start() at :886-935. Kicks off long-running goroutines: handleEvents, acquireNodeLiveness, checkIfReplicasNeedSplitting, updateStoreUsageTag, refreshMetrics, scanReplicas, setupPartitions, plus optional driverQueue and txnCoordinator.
  3. Stop(ctx) at :945-1008. Carefully ordered: stop driver → dropLeadershipForShutdown (transfer leadership of every shard we lead, 8s budget) → stop liveness before leasekeeper → stop the rest → wait 5s so this node can still vote in successor elections → nodeHost.Close() → flush + close Pebble.

Replica: implementing IOnDiskStateMachine

*replica.Replica implements dragonboat's statemachine.IOnDiskStateMachine. There's one Replica per (rangeID, replicaID) hosted by this node. Dragonboat calls it; the Replica reads/writes Pebble.

Pebble layout

One *pebble.IPebbleDB at <rootDir>/raft_store/ is shared by every replica on the node. Keys are organized into prefix tiers: 

\x01..\x02   "local" prefix    — per-replica state, NEVER replicated; replicaID-scoped
\x02..\x03   meta range        — range descriptors; lives only on cluster 1 (MetaRangeID=1)
\x03..\x04   system            — non-RD system data (last IDs, txn records, partitions, sessions)
\x04..       normal data       — splittable user data keyspace

UnsplittableMaxByte = \x04. The splitter refuses split keys below it (see canSplitKeys).

Replica scoping via prefix

When the Replica writes a "local" key, it transparently prepends its replica-specific prefix: 

func LocalKeyPrefix(rangeID, replicaID uint64) []byte {
    name := getName(rangeID, replicaID)           // "c%dn%d"
    prefixString := fmt.Sprintf("%s-", name)
    return append(constants.LocalPrefix, []byte(prefixString)...)
}

So a local key on replica c5n2 is actually stored at \x01c5n2-<key>. This is how multiple replicas of multiple shards share one Pebble DB without colliding.

Ranges & sharding

A range is a contiguous [start, end) byte interval served by one dragonboat shard (one raft group with N replicas). Every range has:

  • RangeId (uint64 — also dragonboat's ShardID)
  • Generation — monotonic, bumped on every descriptor change; used for fencing
  • Replicas (voters), Staging (being added), Removed (being removed) lists
  • Start / End byte slices

The descriptor is stored in two places:

  1. Locally on each replica at LocalRangeKey.
  2. In the meta range at keys.RangeMetaKey(end) = \x02 || end.

This dual storage is why every descriptor change requires a multi-range transaction (local + meta).

Read consistency modes

LINEARIZABLE

Full dragonboat ReadIndex: leader confirms it's still leader before serving. Plus a range-lease check on top.

RANGELEASE

Lease-holder reads from local Pebble without a Raft round-trip. The lease itself guarantees no newer leader could have applied newer writes.

STALE

Skip both checks. Any replica with the range loaded answers from local state. Used for diagnostics and prefetching.

Implementation

Store.SyncRead calls LeasedRange first (range-lease check) then delegates to client.SyncReadLocal, which branches on consistency mode to either nodeHost.SyncRead (ReadIndex) or nodeHost.StaleRead.

The RaftListener

Adapter from dragonboat's IRaftEventListener + ISystemEventListener callbacks to in-process Go channels. Three event types are actively re-published:

EventConsumer
LeaderUpdatedLeaseKeeper (one subscription per shard)
NodeReadyreplicaInitStatusWaiter — knows when each (shard, replica) has loaded
NodeUnloadedLeaseKeeper — stops the per-range agent

Channel sizes are flag-controlled (10 000 / 10 000 / 1 000). Overflow drops are tracked by the RaftListenerEventsDropped Prometheus counter.

Cross-cutting design notes

  • Single Pebble per node, replica-scoped via key prefix. One LSM/WAL/block-cache for all shards on a node; \x01c<rid>n<rep>- namespacing for local state. Replicated data above \x04 is not namespaced and is split based on range boundaries.
  • Two-tier durability. Update commits to Pebble with pebble.NoSync; Sync() (called by dragonboat between batches) does db.LogData(nil, pebble.Sync). The raft log itself is durable in dragonboat's own logdb.
  • Idempotency via sessions. Every BatchCmdRequest may carry Session{Id, Index}. singleUpdate stores the response in SessionPrefix+id and returns the cached response if the entry is re-applied at the same index — important for SyncProposeLocal retries during leader changes.
  • Lease record is replicated state. Unlike many systems where the lease is leader-memory-only, BuildBuddy writes the RangeLeaseRecord into the replicated state machine via CAS. Every voter knows the current lease holder.
  • Generation as a fence. Every range descriptor change bumps generation. Headers carry the expected generation; the replica refuses requests at the wrong generation with RangeNotCurrentMsg, forcing the sender to re-resolve via the meta range.
  • Snapshot format is portable. Not Pebble's native checkpoint or sstable bundle. Manual streaming of length-prefixed proto KVs, with local prefixes stripped so the receiver can re-prefix with its own replicaID.
