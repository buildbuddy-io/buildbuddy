MetadataService server

metadata.Server is the process-level wrapper. It exposes the public MetadataService gRPC and owns the raft Store, the cluster bringup driver, and an asynchronous access-time-update pipeline.

File
enterprise/server/raft/metadata/metadata.go (839 lines). Lives in its own gRPC binary (enterprise/server/cmd/metadata_server/main.go).

The Server struct

metadata.Server owns:

  • A store.Store — the raft node on this machine (NodeHost + replicas + leasekeeper + driver + …).
  • A bringup.ClusterStarter — handles initial cluster formation via gossip.
  • A gossip.Manager — serf discovery for peers.
  • An async access-time-update pipeline (accesses chan *accessTimeUpdate).
  • A filestore.Store — for encoding pebble keys and the storage union (inline / on-disk / GCS).

Construction

Two layers:

  1. NewFromFlags (metadata.go:128-226) — flag-driven entry point. Enforces a default partition (constants.DefaultPartitionID = "default"), creates a gossip.New(nhid) and an optional GCS-backed file storer, assembles a config.ServerConfig.
  2. New (:236-300) — calls store.New (which starts dragonboat NodeHost, opens pebble, starts leasekeeper); calls bringup.New(...) and clusterStarter.InitializeClusters(); spawns processAccessTimeUpdates for batched atime maintenance.

Server.Check (:321-346) is the health-check. Not-ready until cluster bootstrap is done, the store is alive, and s.store.ReplicasInitDone() returns true.

The four gRPC methods

Every method follows the same uniform shape:

  1. Authorize: each FileRecord.Isolation.GroupId must match the caller's group.
  2. Translate: FileRecord → pebble key via fileStorer.PebbleKey(...).Bytes(filestore.Version6).
  3. Wrap into []*sender.KeyMeta (key + caller-supplied metadata).
  4. Fan out via Sender.RunMultiKey — shards by destination range, runs in parallel.
  5. Combine partial responses; for reads, queue atime updates.

Get read

rsps, err := rc.sender().RunMultiKey(ctx, keys, func(ctx, c, h, keys) (any, error) {
    batch := rbuilder.NewBatchBuilder()
    for _, k := range keys {
        batch.Add(&rfpb.GetRequest{Key: k.Key})
    }
    batchProto, err := batch.ToProto()
    rsp, err := c.SyncRead(ctx, &rfpb.SyncReadRequest{Header: h, Batch: batchProto})
    // Per-key: extract GetResponse(i), populate res.found[fr], queue atime update.
}, sender.WithConsistencyMode(rfpb.Header_RANGELEASE))

Note Header_RANGELEASE: the replica holding the range lease can answer locally without going through ReadIndex. For each found record, an atimeUpdateData is queued containing the key, returned atime, and any GCS metadata pointer.

Find read

Same as Get but returns only "present? + atime + gcsMetadata", not full payload. Used by the cache layer for negative-cache checks and atime refreshes.

Set write

metadata.go:718-768
_, err = rc.sender().RunMultiKey(ctx, keys, func(ctx, c, h, keys) (any, error) {
    batch := rbuilder.NewBatchBuilder()
    for _, k := range keys {
        setOp := k.Meta.(*mdpb.SetRequest_SetOperation)
        fm := setOp.GetFileMetadata()
        fm.LastAccessUsec = rc.clock.Now().UnixMicro()
        batch.Add(&rfpb.SetRequest{Key: k.Key, FileMetadata: fm})
    }
    rsp, err := c.SyncPropose(ctx, &rfpb.SyncProposeRequest{Header: h, Batch: batchProto})
}, sender.WithConsistencyMode(rfpb.Header_RANGELEASE))

Writes use SyncPropose (full Raft round-trip). RANGELEASE mode is still used because the leader checks the lease before accepting the proposal — preventing accepted writes from a former leader after partition heal.

Delete admin

Same shape as Set but uses rfpb.DeleteRequest{Key, MatchAtime: deleteOp.GetMatchAtime()}. The optional MatchAtime is the optimistic-concurrency primitive used by the eviction path.

The atime pipeline

Every successful read produces atime data. Updating atime on every read would be punitive; instead, the server batches them.

  1. sendAccessTimeUpdate non-blocking-sends to a buffered channel (atimeBufferSize, default 100k). Buffer full → log and drop.
  2. processAccessTimeUpdates (background goroutine, spawned in New) batches up to atimeWriteBatchSize (default 100) or atimeFlushPeriod (10s).
  3. For GCS-backed entries: first call fileStorer.UpdateBlobAtime to bump the GCS custom-time (so the bucket lifecycle policy won't delete the underlying blob), then propose the atime bump on raft.
  4. The sharded SyncPropose issues UpdateAtimeRequests in a batch via Sender.RunMultiKey.
Order matters here
GCS custom-time bump before raft atime update. If we did it the other way, an eviction sweep could observe the new atime, decide "this is recent, don't delete", but the GCS object would silently TTL out. This way, the worst case is just an unnecessary GCS custom-time bump for a row that was about to be evicted anyway.

Key construction

fileMetadataKey (:365-371) defers to filestore.PebbleKey(fr) and serializes with filestore.Version6. The key embeds isolation (group_id + remote_instance_name), the digest, and the partition.

The partition prefix matters because ranges are partitioned by PT<partition_name>/<hex> — different partitions live in different range groups.

The global keyspace layout is in constants/constants.go:

\x01..\x02   local      — per-replica state, never replicated
\x02..\x03   meta range — range descriptors; never split
\x03..\x04   system     — non-RD system data
\x04..       normal     — splittable user keys

The meta range covers [\x02, \x04) (so meta + system are co-located and unsplittable).

Sender's locality optimization

The MetadataService server doesn't itself have a "skip the network if I'm the leader" shortcut. The optimization happens one layer down in Sender.run / Sender.RunMultiKey:

  1. tryReplicas picks the preferred replica first — the one that last successfully served the request (typically the leaseholder).
  2. If that replica is on the local NodeHost, the gRPC call is over a loopback connection — ultra-low latency.
  3. The real locality bonus is the consistency mode: RANGELEASE means the leaseholder serves reads from local Pebble without a Raft round-trip.

For meta-range descriptor lookups, the bypass is stronger still: Sender.GetMetaRangeDescriptor() reads from the in-process rangeCache seeded by gossip's MetaRangeTag events.

Inline data vs. pointer-to-blob

The metadata server does not know or care. The unit it operates on is sgpb.FileMetadata, which contains a StorageMetadata union (Inline / Pebble / GCS / FileMetadata). What "Set" stores is exactly what was provided by the client.

The only place the metadata server peeks inside StorageMetadata is maybeUpdateGCSAtime (:422-435) — bumping the GCS custom time on atime updates. On Delete, the responsibility for cleaning up the underlying object is not in the metadata server — the eviction path in usagetracker dispatches GCS deletions via a separate worker pool.

The metadata_server binary

enterprise/server/cmd/metadata_server/main.go is a standalone binary that runs one metadata.Server. A production deployment runs N of these (typically 3 or 5) with overlapping partition ownership; they form one logical metadata cluster via gossip + raft.

Each instance:

  • Joins serf gossip with the configured seed addresses.
  • Either bootstraps the initial meta range (if elected by bringup) or rejoins existing shards from on-disk state.
  • Registers MetadataService on its gRPC server.
  • Optionally registers the raft.service.Api gRPC server too (for peer-to-peer routing).
