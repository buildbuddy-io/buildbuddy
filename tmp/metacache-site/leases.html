Range leases & node liveness

A range lease is "I am the authoritative leaseholder for this range right now." It is what lets one replica serve linearizable reads without going through Raft consensus on every read. BuildBuddy implements range leases on top of a separate node-liveness epoch — one renewal per node covers all of its range leases.

Why leases exist on top of Raft

Raft alone is enough for correctness: every read could go through ReadIndex, which confirms leadership before serving. But that's a round-trip per read. A range lease is a longer-term commitment — "no other replica can become leader and apply newer writes until this lease expires" — that lets the leaseholder serve reads from its local Pebble state without a Raft round-trip.

The lease is also the authority for:

  • Proposing membership changes (add / remove / split) for the range.
  • Driving driver-managed work (only the leaseholder evaluates driver actions for its ranges).
  • Issuing range-descriptor updates without an external coordinator.

Three components, three responsibilities

nodeliveness

One record per node. A short-lived lease on (SystemPrefix, nhid) in the meta range. Renewed every ~5s with a 9s TTL. Every range lease this node holds is invalidated when this node's liveness epoch advances.

rangelease

One lease per range. Stored at LocalRangeLeaseKey in the range's own raft group. For normal ranges, anchored to a node-liveness epoch (no separate renewal). For the meta range, anchored to a wall-clock expiration (breaks the circular dependency).

leasekeeper

One agent per range, per store. Subscribes to dragonboat leader-change events and liveness-epoch events; acquires / drops the rangelease accordingly. Bounded-stack-of-1 collapses thrashing leader flips.

Put together

The leasekeeper sees "I'm now the raft leader for range R" → queues "acquire lease R." The acquire CASes LocalRangeLeaseKey with a record anchored to the current liveness epoch. As long as liveness is renewed, the range lease stays valid — no per-range renewal needed.

Renewal — a CAS in the meta range

nodeliveness.go:276 — renewLease
leaseKey := keys.MakeKey(constants.SystemPrefix, h.nhid)
casRequest := rbuilder.NewBatchBuilder().Add(&rfpb.CASRequest{
    Kv:            &rfpb.KV{Key: leaseKey, Value: newVal},
    ExpectedValue: expectedValue,   // previous record
}).ToProto()
rsp, err := h.sender.SyncPropose(ctx, leaseKey, casRequest)

The CAS goes through Sender.SyncPropose — a normal raft write on the meta range. The leaseholder's epoch derives from the local serf.LamportClock, which advances monotonically even on failure (we Witness other leases' epochs to stay ahead of them).

Three outcomes:

  • CAS succeeds → set local record, schedule next renewal at expiration - gracePeriod (≈5s into a 9s lease).
  • CAS fails with FailedPrecondition + CASErrorMessage → another node won. Unmarshal whatever IS at the key, update local record (so next CAS has the right ExpectedValue), bump our Lamport clock past that epoch.
  • Other error → propagate.

Background renewal loop

keepLeaseAlive (:327) is started exactly once via sync.Once after the first successful acquire. Sleeps for timeUntilLeaseRenewal, then forces a renewal. Stops only on context cancel.

Validating other people's range leases

When a replica reads a RangeLeaseRecord.NodeLiveness{nhid, epoch}, it asks the local liveness module: "is this still valid?" via BlockingValidateNodeLiveness:

  1. nhid matches our nhid (we're supposed to be the leaseholder).
  2. We currently have a valid local liveness lease (renew if not).
  3. The range lease's epoch equals our current epoch — if not, this range lease was issued under an old liveness lease and is no longer valid.
Why this is a clever idea
When a node loses its liveness lease (network partition longer than 9s), every range lease it holds becomes instantly invalid cluster-wide. No need to renew thousands of range leases individually. This is the same trick CockroachDB uses ("node liveness epoch").

Range lease

rangelease.Lease represents one range's lease. It is acquired by the raft leader of that range via a CAS write of LocalRangeLeaseKey through the range's own raft group.

Two flavors of lease record

rangelease.go:202-231 — assembleLeaseRequest
if l.rangeID == constants.MetaRangeID {
    // Meta range: wall-clock expiration to break the dependency loop
    // (range lease → node liveness → meta range → range lease …)
    leaseRecord.Value = &rfpb.RangeLeaseRecord_ReplicaExpiration_{
        ReplicaExpiration: &rfpb.RangeLeaseRecord_ReplicaExpiration{

} else {
    // Normal range: anchored to the current node-liveness epoch
    nl, err := l.liveness.BlockingGetCurrentNodeLiveness(ctx)
    leaseRecord.Value = &rfpb.RangeLeaseRecord_NodeLiveness_{ NodeLiveness: nl }
}

Acquisition — sendCasRequest

casRequest, _ := rbuilder.NewBatchBuilder().Add(&rfpb.CASRequest{
    Kv:            &rfpb.KV{Key: leaseKey, Value: newVal},
    ExpectedValue: expectedValue,
}).ToProto()
rsp, err := l.session.SyncProposeLocal(ctx, l.nodeHost, l.replica.RangeID(), casRequest)

The CAS is proposed through the range's own dragonboat shard — so only the raft leader can succeed. If the CAS fails with CASErrorMessage, the current lease is read out of the response and stored locally so the next attempt uses the right ExpectedValue (and the local Lamport clock can be witnessed at the new epoch).

Verification

verifyLease (:138) does three things:

  1. Lease record matches what's currently applied on the replica's state machine.
  2. For node-liveness leases: validate against current liveness epoch.
  3. For time-based leases: check expiration.

Renewal loops

Time-based leases (meta range only) need a periodic renewal goroutine — keepLeaseAlive (:354-372) sleeps (expiration - now) - gracePeriod and re-renews. Node-liveness leases need no per-range renewal — they're transitively renewed every time the node-liveness record is.

Leasekeeper — the dispatcher

One LeaseKeeper per Store manages all of its range-leases. Subscribes to:

  • listener.AddLeaderChangeListener — raft leader changes from dragonboat.
  • listener.AddNodeUnloadedListener — dragonboat replica torn down.
  • liveness.AddListener — this node's liveness epoch changed.

Per-range agent

leasekeeper.go:128-142 — leaseAgent
type leaseAgent struct {
    l       *rangelease.Lease
    updates *boundedstack.BoundedStack    // capacity 1: only latest instruction wins
    ...
}

func (la *leaseAgent) runloop() {
    for {
        instruction, err := la.updates.Recv(la.ctx)
        if err != nil { return }
        la.doSingleInstruction(la.ctx, instruction)
    }
}
Why bounded-stack-of-1?
Rapid leader flapping (e.g. during a network blip) would otherwise queue up many "acquire" + "drop" instructions. With capacity 1, only the latest matters, and the agent doesn't pointlessly process stale ones.

watchLeases — the dispatch loop

EventAction
Raft leader changed for shard R → I am now the local leader, range is openQueue "acquire" for R
Raft leader changed: I am no longer the leader (or range was closed)Queue "drop" for R

AddRange / RemoveRange

Store.AddRange calls leasekeeper.AddRange whenever a new replica is opened (e.g. after Open, or after StartShard). The keeper either creates or loads the agent, marks the range "open", and queues an instruction synchronized with the cached leader status — covering the race where dragonboat's leader-elected callback fired before the range was added.

Self-healing HaveLease

HaveLease(ctx, rangeID) (:474-509) doesn't just return whether the lease is valid — it also queues a corrective instruction if it observes "we should have the lease but don't" or "we shouldn't have it but do." This is a self-healing watchdog called from many code paths.

When the node dies, the renewal stops; after 9s the liveness epoch is considered expired by other nodes; any replica trying to verify our range lease's epoch will fail; some other raft leader (re-elected by dragonboat) will CAS in a new lease record at the next epoch.

Dropping a lease cleanly

dropLease (rangelease.go:118-132) CASes the lease key to empty. Used in two places:

  • LeaseKeeper.RemoveRange — when a replica is torn down (e.g. moved to another node).
  • Store.dropLeadershipForShutdown — during graceful shutdown, after leadership has been transferred.
