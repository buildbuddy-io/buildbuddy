Multi-range transactions (2PC)

Each raft group is independent — there's no cross-group consensus. When an operation must atomically modify multiple ranges (chiefly splits, replica adds/removes, range-descriptor edits), BuildBuddy uses a two-phase commit protocol coordinated by a txn.Coordinator, with a durable TxnRecord in the meta range so crashes don't leave partial state.

File
enterprise/server/raft/txn/txn.go (355 lines).

What needs a transaction

Anything that touches two or more raft groups atomically:

  • Range split: left range descriptor + right range new descriptor + meta range catalog entry.
  • Replica add: this range's descriptor (staging entry) + meta range copy of the descriptor.
  • Replica remove: same — local descriptor + meta range.
  • Admin descriptor updates: local + meta.

Anything that touches one raft group is a normal SyncPropose, no txn needed.

The shape of a transaction

A TxnRequest is {transaction_id, statements[]}; each statement is {range, raw_batch, hooks[], range_validation_required}. Each statement targets exactly one range. See protos for the full shape.

The happy path — RunTxn

  1. Build the proto: generate a UUID transaction_id, assemble statements, wrap in a TxnRecord{TxnRequest, TxnState: PENDING, CreatedAtUsec: now}.
  2. WriteTxnRecord: durably write the TxnRecord to the meta range as a DirectWriteRequest at MakeKey(TxnRecordPrefix, transactionID). Proposed via normal Sender.SyncPropose.
    If we crash now
    The cleanup loop (processTxnRecords) will see the PENDING record and roll back any prepared statements.
  3. Validate: no two statements target the same range (each range can only have one parked batch per transaction).
  4. PREPARE phase: for each statement, run with transaction_id set. Replicas:
    • See the transaction_id.
    • Validate ranges/leases.
    • Write the batch into a per-transaction "prepared" log on disk (LocalTransactionPrefix + txid).
    • Do NOT yet apply the effects.
    prepareError is sticky
    We do not abort on the first error; we still try to prepare the remaining statements so that even on partial failure we have a clear picture of what to roll back.
  5. Decide outcome: 
    operation := rfpb.FinalizeOperation_ROLLBACK
if prepareError == nil {
    operation = rfpb.FinalizeOperation_COMMIT
}
txnRecord.Op = operation
txnRecord.TxnState = rfpb.TxnRecord_PREPARED
if err = tc.WriteTxnRecord(ctx, txnRecord); err != nil {
    return status.WrapErrorf(err, "failed to write txn record")
}
    Writing the PREPARED record commits the decision durably. After this point, even a coordinator crash leads to the same outcome — the cleanup loop sees state=PREPARED and replays the finalize.
  6. FINALIZE phase: for each statement, send a no-op batch with TransactionID=txid, FinalizeOperation=COMMIT|ROLLBACK. The replica un-parks the prepared batch and applies (commit) or discards (rollback). NotFound during ROLLBACK is benign (statement never prepared).
  7. Delete the transaction record. Return prepareError if any.

How each statement is dispatched — Coordinator.run

Because the coordinator already has a RangeDescriptor, it bypasses the meta-range lookup and uses SyncProposeWithRangeDescriptor directly:

txn.go:186
var headerFn header.MakeFunc
if stmt.GetRangeValidationRequired() {
    headerFn = header.MakeLinearizableWithLeaseValidationOnly
} else {
    headerFn = header.MakeLinearizableWithoutRangeValidation
}
retrier := retry.DefaultWithContext(ctx)
for retrier.Next() {
    syncRsp, err := tc.sender().SyncProposeWithRangeDescriptor(ctx, stmt.GetRange(), batch, headerFn)
    if err == nil {
        rsp := rbuilder.NewBatchResponseFromProto(syncRsp.GetBatch())
        if err := rsp.AnyError(); err != nil { return err }
        return nil
    }
    if !status.IsOutOfRangeError(err) && !isConflictKeyError(err) { return err }
    lastError = err
}
Header variants matter
During transactional operations the range's generation can change between the coordinator's snapshot and the replica's application — strict generation matching would always fail. So either skip generation check entirely (new ranges) or only verify the lease.

Crash recovery — processTxnRecords

A background loop runs every txnCleanupPeriod=15s on whichever node currently holds the meta range lease:

txn.go:250
func (tc *Coordinator) processTxnRecords(ctx context.Context) {
    if !tc.store.HasReplicaAndIsLeader(constants.MetaRangeID) {
        return
    }
    txnRecords, err := tc.FetchTxnRecords(ctx, false /*=includeLive*/)
    ...
}

FetchTxnRecords scans TxnRecordPrefix..end, filters out records younger than txnLivenessThreshold=10s (so we don't race a live coordinator), returns the rest.

ProcessTxnRecord:

  • state == PENDING → roll back every statement (some may not have prepared; NotFound is OK).
  • state == PREPARED → replay the recorded Op (commit or rollback) on every statement.
  • Finally delete the record.

This is a saga-style recovery: the durable record is the source of truth, so any node can drive cleanup forward if the original coordinator died.

Why 2PC and not a single multi-key write?

Each raft group is independent — there's no cross-group consensus. The PREPARE phase parks the batches in their respective groups; the PREPARED record in the meta range is the global commit point; the FINALIZE phase is just bookkeeping that re-derives its outcome from the durable record.

This pattern means we can build atomic multi-range splits/merges while keeping each individual raft group simple (it only ever needs to coordinate with itself).

What happens on the replica side

The replica's singleUpdate branches when it sees a transaction_id:

  • PrepareTransaction (replica.go:463) — write batch into prepared[txid], acquire locks on touched keys, persist a LocalTransactionPrefix+txid record for crash recovery.
  • CommitTransaction (:482) — replay the cached batch, delete the txn record.
  • RollbackTransaction (:505) — drop the batch, release locks, delete the txn record.

Key-level locks (lockedKeys) and the special mappedRangeLockingTXID (which locks the entire range) implement isolation. LockMappedRange in the batch (set on split prepares) locks the whole range against any writes — so the split can rewrite the descriptor without losing intervening writes.

The PREPARE-time StartShardHook is what makes the new range actually exist on all members before the commit — by the time the commit applies, the new dragonboat shard is up and the new range descriptor it refers to actually has somewhere to live.
