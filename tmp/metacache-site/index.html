MetaCache & the Raft Metadata Service

A guided, deep tour of enterprise/server/backends/metacache/metacache.go and the multi-Raft cluster behind it (enterprise/server/raft/*). Aimed at engineers who know Raft at the textbook level and want to understand how BuildBuddy actually wires it together with dragonboat, gossip, range leases, and an LSM store.

The big idea, in one paragraph

The BuildBuddy cache splits each blob into metadata (a small FileMetadata proto: digest, partition, isolation, where the bytes live, last-access time, size) and data (the actual bytes). The metadata lives in a strongly-consistent, replicated, sharded key‐value store implemented on top of dragonboat (multi-group Raft), and the data lives in cheap blob storage (GCS) or inline inside the metadata record itself when small. metacache.Cache is the thin glue that presents this to the rest of the system as a normal interfaces.Cache.

Why this design
Bytes are big and write-once; eviction by GCS object-lifetime is cheap. Metadata is small and needs to be strongly consistent (existence, atime, encryption keys). Keeping the metadata in Raft lets the cluster answer "does this CAS digest exist?" linearizably without ever touching GCS, and lets eviction be coordinated across thousands of stores at the metadata layer.

The layers, top-down

1 · MetaCache

The interfaces.Cache implementation. Wraps an mdspb.MetadataServiceClient + a blob store. Handles compression, inlining, encryption, atime.

2 · MetadataService

Public gRPC server inside the raft node. Authorizes, computes pebble keys, fans out via the Sender.

3 · Sender / Registry

Range-aware request router. Caches range descriptors, retries on stale routing, finds gRPC addresses via gossip.

4 · Store + Replica

The dragonboat integration. Store hosts the NodeHost; Replica implements IOnDiskStateMachine per (range, replica).

5 · Range leases & node liveness

Epoch-based leases let the lease-holder serve linearizable reads without quorum. Anchored to a per-node liveness record.

6 · Multi-range 2PC

Splits, replica adds, descriptor edits need atomic updates across multiple raft groups. A durable TxnRecord in the meta range drives crash recovery.

7 · Bringup, driver & splits

How a cluster bootstraps via gossip, how the driver rebalances, how a range splits without dropping writes.

Reading cheatsheet

TermMeans
RangeA contiguous [start, end) byte interval in the global keyspace, served by one dragonboat raft group with N replicas. Every range has a unique range_id (the dragonboat ShardID) and a generation bumped on every descriptor change.
ReplicaOne member of a range's raft group on one node. Identified by (range_id, replica_id). Has a Go *replica.Replica object that implements dragonboat's IOnDiskStateMachine.
NodeHost / NHIDThe dragonboat process-level handle and its stable ID. One NodeHost per node hosts every replica that lives on that node.
Meta rangeRange with range_id=1; stores every other range's descriptor at key \x02 || rd.end. Bootstrap-critical; never split; uses a time-based lease (not node-liveness) to break circular dependency.
Range leaseThe right to serve RANGELEASE-consistency reads locally without a Raft round-trip, and to propose membership changes for the range. Acquired by CAS on a local lease key; anchored to the holder's node-liveness epoch.
SenderThe thing that turns "do this batch on key K" into "find the right range, find a willing replica, do the RPC, retry if stale." Lives client-side in every node that issues raft traffic.
DriverPer-node background process that scans local leased replicas and enqueues actions (split, add replica, remove dead replica, transfer lease) based on cluster-wide load.
StorageMetadataDiscriminated union inside a FileMetadata saying where the bytes live: InlineMetadata (≤ ~1 KiB, bytes embedded), GCSMetadata (object name + custom-time), or PebbleMetadata/FileMetadata (used by other backends, not by metacache).
