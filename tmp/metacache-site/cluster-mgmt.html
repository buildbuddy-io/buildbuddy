Bringup, driver, splits & usage

How a fresh cluster bootstraps, how the placement driver keeps things balanced, how a range splits without downtime, and how partition usage is tracked across the fleet.

Bringup — initial cluster formation

enterprise/server/raft/bringup/bringup.go (612 lines).

There is no static config file of cluster members. The cluster forms via serf gossip, seeded by a flag-driven JoinList.

  1. NewFromFlags creates a gossip.New(nhid) with the configured raft-seed pod hostnames as the join list.
  2. bringup.New(grpcAddr, gossipMan, store, partitions) constructs a ClusterStarter. Only nodes whose own listen address matches one of the join-list entries register as gossip listeners eligible for auto-bringup: 
    for _, joinAddr := range cs.join {
    if inJoinList, err := cs.matchesListenAddress(joinAddr); err == nil && inJoinList {
        gossipMan.AddListener(cs)
        break
    }
}
    This makes the "bringup quorum" the join-list nodes.
  3. InitializeClusters decides whether to bootstrap:
    • If store.ConfiguredClusters() > 0 (pebble has previous replicas), skip bootstrap; store.Start rejoins them automatically.
    • Otherwise, if this node matches the first join-list entry (the elected bringup coordinator), kick off a goroutine that periodically calls attemptQueryAndBringupOnce.
  4. attemptQueryAndBringupOnce issues a serf Query named AutoBringupEvent. Every node listening replies with {nhid, grpcAddress}. The coordinator collects responses until len(bootstrapInfo) == len(cs.join) — heard from every seed node — then:
    • InitializeShardsForMetaRange(ctx, session, store, bootstrapInfo) — bootstraps the single meta range with rangeID = MetaRangeID = 1.
    • InitializeShardsForPartition(ctx, store, bootstrapInfo, defaultPartition) — creates the first partition's shards.

Meta-range initialization

InitializeShardsForMetaRange sorts NHIDs deterministically, assigns sequential replica IDs, issues apiClient.StartShard in parallel against every node (dragonboat needs the same InitialMember map on every node for leader election to converge), then proposes one batch that:

  • Writes LocalRangeKey → range descriptor (per-replica local key).
  • Writes ClusterSetupTimeKey.
  • Writes RangeMetaKey(rd.End) → the meta range's own descriptor (so the meta range contains its own descriptor at the key \x02 + UnsplittableMaxByte).
  • Increments LastRangeIDKey so the next range gets ID 2.
  • Increments LastReplicaIDKeyPrefix + "1" by len(replicas).

Default partition initialization

InitializeShardsForPartition (bringup.go:463-612):

  1. evenlyDividePartitionIntoRanges chops the 32-byte hex digest space [0, 2^256) into partition.NumRanges equal-width ranges, each keyed by PT<partition_id>/<hex>.
  2. Reserves new range IDs via store.ReserveRangeIDs(n) (an Increment on LastRangeIDKey).
  3. Writes a PartitionDescriptor (state INITIALIZING) to the meta range first. Makes the operation crash-recoverable.
  4. For each new range, calls MakeBootstrapInfo + startShard in parallel via errgroup.
  5. Builds a single transaction (store.TxnCoordinator().RunTxn) that atomically writes the local range key + cluster-setup time on every new shard, increments the last-replica-id counter for each new range, and CAS-flips the partition descriptor to INITIALIZED.

Join existing vs bootstrap new

ScenarioWhat happens
Existing pebble has replicas (ConfiguredClusters() > 0)Bringup skipped. store.Start iterates NodeHostInfo.LogInfo and calls StartOnDiskReplica for each, which rejoins the existing raft membership.
Fresh pebble, this is the elected bringup coordinatorWait for all join-list nodes via gossip AutoBringupEvent; then bootstrap meta range + default partition.
Fresh pebble, this is NOT the coordinatorListen for incoming StartShard RPCs from the coordinator.

Driver — the placement engine

enterprise/server/raft/driver/driver.go (1 683 lines).

The driver is a priority-queued background scheduler running on every store. It is fed two kinds of tasks: RangeTaskType (per rangeID) and PartitionTaskType (per partitionID).

Actions, ranked by priority

ActionPriorityWhen
DriverInitializePartition800A configured partition has no shards yet
DriverReplaceDeadReplica700Replica's host is dead and we're at min replication
DriverAddReplica600Below minReplicasPerRange (default 3)
DriverFinishReplicaRemoval500Leftover pebble data from a previously-removed replica
DriverRemoveDeadReplica400Dead replica beyond min replication
DriverRemoveReplica300Above min replication, healthy
DriverSplitRange200Range size ≥ target_range_size_bytes (default 100MB)
DriverRebalanceReplica0Source is above-mean, dest is below-mean
DriverRebalanceLease0Co-located replica with much lower lease count

Health-critical actions preempt rebalancing.

What fills the queue

  • Store.scanReplicas — periodic ticker enumerates leased replicas; for each, driverQueue.MaybeAddRangeTask.
  • Store.checkIfReplicasNeedSplitting — listens on EventRangeUsageUpdated; when a leased range crosses a size watermark, immediately enqueues (faster than waiting for the next scan).
  • Store.ProcessPartitions — meta-range leader periodically checks for partitions needing bringup.
  • After applyChange succeeds: task is re-enqueued with RequeueCheckOtherActions so subsequent imbalances get re-evaluated.

Retry & thrash avoidance

  • attemptRecord stores (action, attempts, nextAttemptTime).
  • On RequeueRetry: nextAttemptTime = now + 1s * 2^attempts (exponential backoff).
  • After maxRetry=10 attempts, drop the task; next periodic scan re-enqueues from scratch if still needed.
  • process compares the current needed action to the previously-attempted one; if same action is still needed but inside backoff window, do nothing.
  • Queue capped at maxSize=1000; when overflowed, lowest-priority item is dropped.

Decision logic — computeActionForRangeTask

Runs only if the local node holds the lease for the range. Order of conditionals = order of decisions:

  1. If rd.GetDeleted() → noop.
  2. If len(rd.Removed) > 0 → tentatively set FinishReplicaRemoval; continue.
  3. curReplicas < minReplicas OR len(rd.Staging) > 0 → AddReplica; priority boosted by (desiredQuorum - curReplicas).
  4. If numLiveReplicas < quorum → only FinishReplicaRemoval is safe (can't change membership without quorum).
  5. If curReplicas <= minReplicas && numDeadReplicas > 0 → ReplaceDeadReplica.
  6. If numDeadReplicas > 0 (over-replicated) → RemoveDeadReplica.
  7. If curReplicas > minReplicas → RemoveReplica.
  8. Meta range → noop (never auto-rebalance).
  9. If cluster healthy and split enabled → check size: 
    jitter := (rand.Float64()*2 - 1) * jitterFactor * float64(targetRangeSizeBytes)
threshold := float64(targetRangeSizeBytes) + jitter
if usage.GetEstimatedDiskBytesUsed() >= threshold {
    action = DriverSplitRange
}
    Jitter prevents thundering-herd splits across replicas.
  10. Cluster-healthy rebalance: findRebalanceReplicaOp looks for above-mean source store, below-mean destination.
  11. Lease rebalance: findRebalanceLeaseOp — first checks storeMap.CheckLeaseRebalancePrecondition (all stores Available & Ready; cluster stable for 5min; ranges-without-leases below threshold).

Executing changes — applyChange

A change is a tuple of optional ops:

type change struct {
    addOp                *rfpb.AddReplicaRequest
    removeOp             *rfpb.RemoveReplicaRequest
    removeDataOp         *removeDataOp
    splitOp              *rfpb.SplitRangeRequest
    transferLeadershipOp *rfpb.TransferLeadershipRequest
}

Applied in fixed order: split → add → remove → removeData → transferLeadership. For ReplaceDeadReplica the driver builds {addOp, removeOp}; add succeeds first so the cluster still has quorum, then the remove can drop a now-redundant dead replica.

Range splits — end to end

A split takes one range and turns it into two — the original ("left") covers the first half, the new ("right") gets the second half. The data is not moved; both ranges live on the same set of stores initially. Migration happens later if/when the driver rebalances replicas of the right range.

Trigger

Two sources, both feeding driverQueue.MaybeAddRangeTask:

  • Store.checkIfReplicasNeedSplitting: RangeUsageEvent with estimatedDiskBytes ≥ (1-jitter) * targetRangeSizeBytes.
  • Periodic Store.scanReplicas.

Decision

Driver returns DriverSplitRange only when: this node holds the lease, cluster is healthy (AllStoresAvailableAndReady), split enabled, size threshold met.

Execution — Store.SplitRange

  1. Pre-flight: assert we have the lease (HaveLease); validate the descriptor against the meta range (catches a stale local view).
  2. Reserve IDs: ReserveRangeID for newRangeID (Increment on LastRangeIDKey); reserveReplicaIDs(ctx, newRangeID, len(replicas)).
  3. Find split point: SyncReadLocalBatch with FindSplitPointRequest against the local leader. Replica.findSplitPoint iterates pebble between [rd.Start, rd.End), computes total size, picks the rowkey closest to 50% byte point subject to constraints (>2 rows on each side, between 20% and 80% of size, respects canSplitKeys).
  4. Build the transaction — BuildTxnForRangeSplit constructs three statements (atomic across three ranges):
    Left statement (existing range)
    • CAS-update LocalRangeKey with shrunk End=splitKey, Generation+1.
    • LockMappedRange=true — during prepare, leader rejects writes to the second half (no lost writes).
    • Post-commit hooks:
      • On PREPARE: StartShardHook{NewRangeID, InitialMember} — every replica spawns a new dragonboat raft group for the right range on the same node.
      • On COMMIT: SnapshotClusterHook — take a local snapshot to compact the now-shrunk left log.
    Right statement (new range) Direct-write a LocalRangeKey with the new descriptor on the new range's raft group. Post-commit SnapshotClusterHook.
    Meta statement (meta range) Two CAS ops — insert the right descriptor at RangeMetaKey(NewRight.End), update the left descriptor at its new RangeMetaKey(Left.End). Validates atomically that no one else changed the descriptors.
  5. Run the transaction: txnCoordinator.RunTxn(ctx, splitRangeTxn.TxnBuilder). The 2PC protocol persists a TxnRecord in the meta range so a crashed coordinator can resume — see Transactions.

Online?

Yes, with a brief lock window. During PREPARE the left range's leader stops accepting writes to the right half (LockMappedRange=true). Once COMMIT applies, writes to the right half are addressed to the new range. The lock window is on the order of the txn duration (tens of milliseconds in steady state). Reads keep working throughout because they go through the lease.

Zero data movement at split time
The pebble bytes don't move — both ranges share the same store-local pebble instance, just with different [start, end) bounds. A split is cheap on the local node; the only network cost is the txn coordination and starting a new dragonboat group (which immediately gets a snapshot of the relevant subspace).

Replica moves (add + remove)

Two flavors:

  • Rebalance: move replica from store A to store B (both healthy).
  • Replace dead: same shape, but the "from" replica is on a dead store and can't gracefully transfer state.

Driver constructs the change in rebalanceReplica:

return &change{
    addOp: &rfpb.AddReplicaRequest{ Range: rd, Node: op.to.usage.GetNode() },
    removeOp: &rfpb.RemoveReplicaRequest{ Range: rd, ReplicaId: replicaID },
}

applyChange runs add → remove (so quorum isn't broken).

AddReplica

Multi-step state machine, recoverable on retry from raft membership + the staging-replicas list:

  1. Validate: non-empty replicas, target node not already a member, no overlapping range on target. Confirms the local lease and validates the supplied range descriptor against the meta range.
  2. Reserve replica ID + add to Staging. reserveReplicaIDs(ctx, rangeID, 1) increments the per-range last-replica counter; addStagingReplicaToRangeDescriptor writes the new replica into rd.Staging (NOT yet a voter). Both local + meta-range descriptors updated atomically.
  3. Add as non-voter: get ConfigChangeID, then nodeHost.SyncRequestAddNonVoting(ctx, rangeID, newReplicaID, nhid, configChangeID). Non-voter receives state but doesn't count toward quorum — safe during catch-up.
  4. Wait and promote to voter:
    • getLocalLastAppliedIndex of the current leader.
    • apiClient.Get(node.GrpcAddr).StartShard(...) on the new node with Join=true, IsNonVoting=true, LastAppliedIndex=.... The remote StartShard calls nodeHost.StartOnDiskReplica and then blocks via waitForReplicaToCatchUp until applied index ≥ LastAppliedIndex.
    • nodeHost.SyncRequestAddReplica(...) to promote non-voter to voter.
  5. Move from Staging to Replicas: addReplicaToRangeDescriptor; updates local + meta descriptor.

RemoveReplica

  1. Lease check; cannot remove the local leader (driver works around this by first transferring leadership).
  2. markReplicaForRemovalFromRangeDescriptor: moves the replica from rd.Replicas into rd.Removed. The data is not yet deleted — this is the "tombstone" so a subsequent DriverFinishReplicaRemoval can clean up.
  3. syncRequestDeleteReplica(ctx, rangeID, replicaID): config-change to remove from dragonboat membership.

FinishReplicaRemoval — data wipe

Driver constructs change{removeDataOp:...}; applyChange contacts the removed node:

c, _ := rq.apiClient.GetForReplica(ctx, op.replDesc)
_, err = c.RemoveData(ctx, &rfpb.RemoveDataRequest{
    ReplicaId: op.replDesc.GetReplicaId(),
    Range:     op.rd,
})

This deletes pebble data for that range and removes the replica from rd.Removed. (Driver checks storeMap.DivideByStatus first; only fires when the dead node is reachable. If truly dead, the data is orphaned and reclaimed by a janitor on next restart.)

Why staging exists

If AddReplica crashes between "non-voter added" and "promoted to voter", the next attempt sees a staging entry and a matching non-voter in raft membership, and resumes promotion rather than reserving a new replica ID.

TransferLeadership

Trivial: s.nodeHost.RequestLeaderTransfer(rangeID, targetReplicaID). Driver invokes for DriverRebalanceLease and (implicitly) before removing the local-leader replica. After transfer, the dragonboat leader-change event fires; both old and new leaders' leasekeepers react.

Usage tracker & eviction

enterprise/server/raft/usagetracker/usagetracker.go (922 lines).

Tracks per-partition disk usage with two views:

  1. Local (partitionUsage.sizeBytes): bytes this store consumes for keys in this partition. Sourced from db.EstimateDiskUsage(start, end) (pebble built-in), refreshed every 10s.
  2. Global (partitionUsage.nodes): every other store's most-recent reported size for this partition. Aggregated by GlobalSizeBytes() as a sum.

Global propagation

Via serf gossip user-events:

  • broadcastLoop runs two timers: a check every 15s, and an "always broadcast" idle timer after 5 minutes of silence.
  • broadcast(force): skip if no partition's size changed by >100MB (the gossip delta threshold). Else marshal NodePartitionUsage proto, send NodePartitionUsageEvent.
  • Receivers' OnEvent updates byPartition[id].nodes[nhid].sizeBytes and feeds the per-partition approxlru.LRU.

Eviction pipeline

Each partition runs four goroutines:

  1. startSampleGenerator: forward-scans pebble in random-key order, yielding samples (key + atime + size + storage_metadata) into pu.samples channel. Refreshes the iterator every 5 minutes.
  2. processEviction: pulls samples that approxlru decided to evict, batches up to evictionBatchSize, sends DeleteRequest{Key, MatchAtime} batches via sender.RunMultiKey.
  3. processGCSDeletions: for evicted entries whose storage was GCS, calls fileStorer.DeleteStoredBlob.
  4. updateLocalSizeBytes: as above.

The approxlru.LRU is constructed with MaxSizeBytes = 0.90 * partition.MaxSizeBytes, so eviction kicks in at 90% full. Rate-limited (evictionRateLimit=300/sec).

MatchAtime is the safety mechanism
The eviction sampler observes a row's atime; DeleteRequest{Key, MatchAtime} only deletes if that atime hasn't changed. If anyone touched the row between sample and delete (atime updates flow through the metadata service's atime pipeline), the delete fails with FailedPrecondition and the row survives.

Why gossip + meta range together?

The meta range is the durable source of truth for range descriptors and partition descriptors. Gossip is the cache-warming channel. Without gossip, every node would have to ReadIndex the meta range on every cache miss; with gossip, range descriptors arrive at every node within seconds of being written.

Critically: gossip is allowed to be wrong; the meta range is not. If gossip lies, the worst outcome is a wasted RPC attempt that gets OutOfRange from the wrong-generation replica, and the sender re-fetches from the meta range.
