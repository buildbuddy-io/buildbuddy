End-to-end walkthroughs

Concrete traces of representative requests, from the network edge all the way down to a Pebble commit (or GCS PUT), and back up. Switch between scenarios with the tabs.

Scenario A — Bazel uploads a 2 MiB CAS blob

The classic case: large enough to go to GCS, with auto-zstd.

  1. Bazel calls ByteStream.Write with the blob bytes against the app server.
  2. app server / cache server receives the streaming request. env.GetCache() returns the registered metacache.Cache. The cache's Writer(ctx, resourceName) is called.
  3. metacache.Cache.Writer → writerWithImmediateCommit → writer:
    • The size hint (digest size, ~2 MiB) is > MaxInlineFileSizeBytes (1024) → blob path.
    • The size hint is also ≥ MinBytesAutoZstdCompression (100) → zstd-compress.
    • The ResourceName is rebuilt with Compressor: ZSTD so the FileRecord records the on-disk compression.
  4. writerForRecord returns filestore.BlobWriter(ctx, fileRecord). Inside:
    • blobName = {appName}/PART_default/GR123/cas/abcd/abcd123… + random 5-char salt.
    • gcs.ConditionalWriter(ctx, blobName, overwrite=true, customTime=now, estimatedSize=size/5).
  5. Optional encryption wrap if the group has encryption enabled (encryptionEnabled checks authutil.EncryptionEnabled + env.GetCrypter()).
  6. zstd wrap: compression.NewZstdCompressingWriteCommiter with 4 MiB buffer from the shared pool.
  7. Bazel's bytes flow: client → ByteStream → write chain → zstd encoder → (optional encryptor) → GCS streaming writer.
  8. wc.Commit() propagates down: zstd flushes (records compression-ratio metric), encryptor flushes, GCS conditional writer commits the object.
  9. Commit callback fires: builds the FileMetadata: 
    md := &sgpb.FileMetadata{
    FileRecord: fileRecord,
    StorageMetadata: &sgpb.StorageMetadata{
        GcsMetadata: &sgpb.StorageMetadata_GCSMetadata{
            BlobName: "…cas/abcd/abcd…-XK4z2",
            LastCustomTimeUsec: now,
        },
    },
    StoredSizeBytes: bytesWritten,  // ~600 KiB after zstd
    LastAccessUsec: now,
    LastModifyUsec: now,
}
  10. setMetadatas(ctx, md) → MetadataService.Set gRPC over the configured backend address.
  11. metadata.Server.Set:
    • Authorize the group_id.
    • Compute pebble key via fileStorer.PebbleKey(fr).Bytes(Version6).
    • Sender.RunMultiKey(keys, fn, RANGELEASE) with one key.
  12. Sender.partitionKeysByRange:
    • RangeCache lookup for key → typically a hit (range descriptor cached from prior traffic / gossip).
    • One bucket, one key.
  13. tryReplicas(rd, fn):
    • Tries rd.Replicas[0] (the preferred replica = leaseholder from last successful call).
    • apiClient.GetForReplica(replica) → registry.ResolveGRPC(nhid) → cached gRPC pool.
    • Build header with RANGELEASE + this range's generation.
    • Issue SyncPropose(SyncProposeRequest{Header, Batch}).
  14. Remote Store.SyncPropose:
    • LeasedRange(ctx, header): replica present? lease held? generation matches? yes/yes/yes.
    • Pick the right session (regular write session), call session.SyncProposeLocal(ctx, nodeHost, rangeID, batch).
  15. Dragonboat consensus:
    • Leader appends entry to local raft log, replicates to followers, waits for quorum acks, commits.
    • On every voter, dragonboat calls Replica.Update(entries).
  16. Replica.singleUpdate:
    • Decode BatchCmdRequest; check idempotency session (first time, no cached response).
    • Validate Header.Generation against current range descriptor.
    • Iterate RequestUnions: one SetRequest → handlePropose → set:
      • filestore.PebbleKey.Validate the key.
      • proto.Marshal the FileMetadata.
      • wb.Set(key, value, pebble.NoSync) — written into the indexed pebble batch.
    • commitIndexBatch(wb, entry.Index) — atomically writes LastAppliedIndexKey = entry.Index into the same batch and wb.Commit(pebble.NoSync).
    • Marshal BatchCmdResponse into entry.Result.Data.
  17. Dragonboat Sync (in the background, between batches): db.LogData(nil, pebble.Sync) — fsyncs the WAL.
  18. Response unwinds: SyncProposeLocal returns → Store.SyncPropose returns → gRPC response → Sender.run returns → metadata.Server.Set returns SetResponse{} → metacache.setMetadatas returns nil → Commit() returns success → ByteStream returns success to Bazel.
Failure modes
  • GCS write fails → Commit() returns error; no metadata written; no orphan.
  • GCS succeeds, metadata fails → blob is orphaned in GCS; TTL'd out after GCSTTLDays.
  • Lease holder dies mid-write → Sender retries on next replica; once a new lease is established, the write completes.
  • Same blob written twice concurrently → two distinct GCS objects (different salts); metadata's last write wins; the loser's blob is orphaned.

Scenario B — Bazel uploads a 200 byte CAS blob

Below the 1024-byte inline threshold and below the 100-byte zstd threshold. Smallest path.

  1. Same start: Bazel → ByteStream.Write → metacache.Cache.Writer.
  2. writer:
    • Size hint 200 > 100 → would compress, BUT size hint < 1024 → inline.
    • Actually compression happens before the writer is picked; for inline below-threshold writes, compressed bytes would still go inline. In this case the configured MinBytesAutoZstdCompression default is 100 so it DOES compress; but if your record is < 100 bytes it stays IDENTITY.
  3. writerForRecord: sizeHint 200 < 1024 → returns filestore.InlineWriter(ctx, 200), a bytes.Buffer-backed writer.
  4. Write bytes: flow through zstd → into bytes.Buffer.
  5. Commit(): nothing to flush externally; the wrapper's CommitFn fires: 
    md := &sgpb.FileMetadata{
    FileRecord: fileRecord,
    StorageMetadata: &sgpb.StorageMetadata{
        InlineMetadata: &sgpb.StorageMetadata_InlineMetadata{
            Data: bytesBuffer.Bytes(),       // the (compressed) bytes themselves
            CreatedAtNsec: writtenAt.UnixNano(),
        },
    },
    StoredSizeBytes: 80,    // ~80 bytes after zstd
    LastAccessUsec: now,
    LastModifyUsec: now,
}
  6. setMetadatas → MetadataService.Set → Sender → SyncPropose → Replica.Update.
  7. The bytes ride along inside the FileMetadata record, written as a single pebble KV by the replica. No GCS write happens at all.
Why this matters
Most CAS entries are small (file digests, AC entries). Inlining means thousands of writes/sec can complete with just a single raft consensus round and zero GCS operations. The metadata cluster carries the data.

Scenario C — Bazel reads a CAS blob

  1. Bazel calls ByteStream.Read.
  2. app server → metacache.Cache.Reader(ctx, r, 0, 0).
  3. Reader:
    • makeFileRecord for the resource.
    • lookupMetadatas → MetadataService.Get → Sender.RunMultiKey → SyncRead(RANGELEASE).
    • Leaseholder serves from local pebble without a raft round-trip (RANGELEASE consistency).
  4. internal reader(md, ...):
    • Zero-length guard.
    • If GCS: ObjectIsPastTTL pre-check with 1h buffer.
    • If encrypted but encryption disabled now → NotFound.
    • Decide rawStorage: if cached is IDENTITY and no decryption needed, push offset/limit to the backend.
  5. open backend reader:
    • For inline: bytes.NewReader(im.Data[off:][:lim]).
    • For GCS: streaming reader with byte-range request.
  6. Wrap chain: decryptor (if encrypted) → decompressor (if cached ZSTD, requested IDENTITY) → offset skip → limit. If client wants ZSTD and cached is IDENTITY, compress on read instead.
  7. Stream back to Bazel; io.Copy through the chain.
  8. Atime update: metadata.Server.Get queued an atimeUpdateData; background processAccessTimeUpdates will batch and propose UpdateAtimeRequest later (and bump GCS custom-time first if applicable).
If the GCS blob is gone
The backend reader returns NotFound. handleMetadataMismatch issues a conditional DeleteRequest{MatchAtime: md.LastAccessUsec} to clean up the metadata. If a concurrent writer wrote a fresher record, the delete fails with FailedPrecondition and the new record stays.

Scenario D — Bazel calls FindMissingBlobs with 500 digests

The bulk existence-check path. Pure metadata; no GCS at all.

  1. Bazel → ContentAddressableStorage.FindMissingBlobs(digests[]).
  2. app server → metacache.Cache.FindMissing(resources).
  3. FindMissing:
    • Build 500 FileRecords.
    • Single MetadataService.Find RPC.
  4. metadata.Server.Find → Sender.RunMultiKey(keys=500, fn, RANGELEASE).
  5. Sender.partitionKeysByRange:
    • For each key, rangecache lookup. With a steady-state cache, mostly hits.
    • Misses (e.g. after a split) → batched into one meta-range scan.
    • Group keys by range_id. Say 500 keys span 5 ranges → 5 buckets.
  6. Fan out: errgroup of 5 goroutines, each issuing one SyncRead(FindRequest×N) against the leaseholder of its range.
  7. Per range: leader's Replica processes the batch in handleRead; each FindRequest becomes a pebble Get; result is a FindOperationResponse{present, last_access_usec}.
  8. Combine: metadata.Server.Find merges the 5 partial responses into one ordered FindResponse matching the request's slot order.
  9. metacache.FindMissing filters where present=false, returns the missing digests.
  10. app server returns to Bazel.

500 digests, 5 ranges, 5 parallel SyncRead RPCs, no GCS touches, no raft writes. The atime updates feed the asynchronous pipeline for later batching.

Scenario E — Sender hits a stale range descriptor

A range just split. Sender's cached descriptor points at the old (pre-split) range.

  1. Sender.run for some key K. Rangecache hit on the stale descriptor RDold (covered [A, Z), but the actual current state is [A, M) and [M, Z), with K in [M, Z)).
  2. tryReplicas(RDold, fn):
    • Tries replica 0. The receiving Store's LeasedRange validates header.RangeId == 1234 and header.Generation == 7. But the local descriptor on that node has generation 8 now (post-split). Mismatch. Returns RangeNotCurrentMsg.
  3. Sender.tryReplica recognizes RangeNotCurrentMsg: this is a generation mismatch, no point trying other replicas (they all have the same generation). Bails the whole tryReplicas loop.
  4. Sender.run loop: err is an OutOfRangeError. Sets skipRangeCache = true; retries.
  5. Next iteration:
    • LookupRangeDescriptor(ctx, key, skipCache=true).
    • Bypasses cache; calls fetchRangeDescriptorsFromMetaRange([key]).
    • Meta range scan returns the fresh RDnew2 (covering [M, Z), generation 1).
    • rangeCache.UpdateRange(RDnew2) — since generation is from a different range_id this is a separate entry, replacing the stale [A, Z) entry via AddAndRemoveOverlapping.
  6. tryReplicas(RDnew2, fn): now correct. Tries replica 0; success.
  7. SetPreferredReplica: replica 0 was already preferred; no reorder.
  8. Result returned.
Self-healing
The first stale request pays the cost (one wasted RPC + one meta-range scan). Subsequent requests see the fresh descriptor in cache. The cache is never invalidated explicitly — it just gets bypassed for one operation, and the fresh descriptor's generation makes it the new entry.

Cascading variants

  • Multi-key (RunMultiKey): partitionKeysByRange detects two keys mapping to the same range_id returning non-equal RDs and retries the whole partitioning. Each retry refetches missing descriptors.
  • Lease just transferred: receiver returns RangeNotLeasedMsg instead. tryReplica returns true (continue to next replica) rather than bailing. After a few replicas the new leaseholder is found and promoted via SetPreferredReplica.
  • Node restarted, lost liveness epoch: any replica that was holding a range lease keyed to the old epoch returns RangeLeaseInvalidMsg → continue to next replica. The replica that gets elected raft leader by dragonboat will acquire a fresh lease with the new epoch on its next leasekeeper tick.
