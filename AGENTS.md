# buildbuddy

## Building and testing code

- This is a bazel project - use `bazel build` to ensure that targets compile.
- This bazel project supports remote execution and remote caching.
  After making large changes, you can efficiently test the entire repo using
  `bazel test //... --config=remote` which runs tests using BuildBuddy's
  remote caching and remote execution. This command is *roughly* equivalent
  to what we run on CI (except in CI the race detector is enabled).
- Tests that have `tags = ["docker"]` in the bazel test rule MUST be run
  with `--config=remote` since they each run on their own remote VM with an
  ephemeral `dockerd` instance.
- When iterating on a single test case, use `--test_filter` to run just that test.
- To run executables, make sure to use `bazel run`, to ensure they are freshly
  built before running. The syntax is:
  `bazel run [bazel_flags] -- //path/to/binary [binary_args...]`

## Dependency management

- There can be no dependencies on `//enterprise/...` subpackages from any
  other packages. The buildbuddy-foss repo (fully open source version)
  is generated by simply deleting the `//enterprise` directory, so this
  is a hard requirement. So, do NOT try to add visibility rules in order to
  allow exceptions here - it will break the buildbuddy-foss repo!

## Code style

### General

- Avoid introducing tiny wrapper/helper funcs (i.e. avoid excessively "DRY" code),
  except in cases where the helpers are actually called several times
  or it significantly improves readability (e.g. it helps remove multiple levels of nesting).
  Don't try to "predict" whether a function will be reused; instead prefer to DRY up the code during review/cleanup passes.

### Golang

- Always format code using `goimports`, not `gofmt`.
- For proto imports, alias protos like `xxpb "<proto_package>"`
  where `xx` is a short abbreviation for the package basename.
  Make sure to follow repo conventions - e.g. the "remote_execution"
  proto is conventionally imported as `repb`.
  Not all code uses these conventions but make sure to follow
  this convention anyway for newly added imports.
- Always alias imports for packages that are just versions like "v2"
  or which have "-go" suffixes, e.g. "sendgrid-go" could be aliased
  as "sg".
- Do NOT alias other imports unless there is a conflict.
- When wrapping errors, use a terse style, avoiding "failed" / "error".
  Example:
```go
if err := doThing(); err != nil {
        return fmt.Errorf("do thing: %w", err)
}
```
- Do NOT redeclare the loop variable like `for _, x := range foo { x := x; ... }`
  This is NOT NEEDED as of Go 1.22, which changed the scoping rules for the loop
  variable so that each loop iteration gets its own scope.
- Prefer using the new loop syntax where applicable:
```go
// Do thing ten times
for i := range 10 {
  doThing(i)
}
```
or
```go
// Do thing ten times
for range 10 {
  doThing()
}
```
- Keep flag definitions (e.g. flag.Bool) on one line. Inline flag default values instead of using consts.
- When working protos (e.g. gRPC requests/responses), avoid nil checks when possible.
  Note that the protobuf codegen for Go allows things like `someProto.GetSomeField()` to not panic, even if `someProto == nil`.
  The generated `GetSomeField()` will return the zero value for the field if the pointer receiver is nil.

#### Golang - tests

- Use `stretchr/testify` for test assertions.
- Sprinkle comments in the test body that tell the story of what scenario we're testing
  as well as the outcome we're expecting. The narrative should be self contained;
  comments should not implicitly reference comments from other tests via words like "same", "still", etc.
- Do not use `t.Helper()` except for pure assertion helpers that don't do I/O
  (e.g. `requireListSorted`, `assertAllValuesLowercase()`, etc.)
- Do NOT create new "internal tests" in the same Go package if an existing test already exists.
  For example if `foo.go` and `foo_test.go` already exist, do NOT try to introduce
  `foo_internal_test.go` just to get access to the `foo` package.
  Instead, converse with me and we'll find a better solution.
